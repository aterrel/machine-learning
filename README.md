# Agentic Project Starter

A ready-to-use multi-agent development workflow for software projects.
Fill in one file. Run one command. Start building with a coordinated team of AI agents.

---

## How It Works

This starter provides a **multi-agent workflow** where specialized AI agents collaborate on your project:

- **Claude Manager** — Orchestrates everything. Runs `/go` and `/bootstrap`.
- **Product Manager** — Translates your objectives into detailed requirements (REQ documents).
- **Software Architect** — Designs the system architecture (ARCH documents).
- **Programmer** — Implements features per the REQ + ARCH documents.
- **QA Agent** — Writes test plans and integration tests.
- **Tech Lead** — Reviews all sprint output before the sprint closes (mandatory gate).
- **Project Manager** — Tracks sprint progress, maintains PROJECT_STATUS.md.

Agents communicate through structured documents (REQs, ARCHs, handoffs) stored in `agents/`.
Git history is the audit trail — every agent commit uses a prefix like `[Prog]` or `[Arch]`.

## Getting Started (5 steps)

### Step 1: Copy this directory

```bash
cp -r agentic-project-starter/ my-new-project/
cd my-new-project/
git init
git add .
git commit -m "Initialize project from agentic-project-starter"
```

Or use it as a GitHub template if you've pushed it there.

### Step 2: Fill in `PROJECT.md`

Open `PROJECT.md` and fill in:
- **Project Name** — What is this called?
- **Vision** — 1-3 sentences on what it does and for whom
- **High-Level Objectives** — 3-8 user-facing outcomes (these become your REQ documents)
- **Tech Stack** — Optional. Leave blank for AI to recommend.
- **Constraints** — Offline-first? Single developer? Privacy requirements?
- **Success Metrics** — How will you know it worked?

This is the only file you *must* fill in before starting.

### Step 3: Run `/bootstrap` in Claude Code

Open Claude Code in your project directory:
```bash
claude
```

Then type:
```
/bootstrap
```

Claude Manager will:
- Read your PROJECT.md
- Generate a project-specific CLAUDE.md
- Create all agent infrastructure directories
- Write PROJECT_STATUS.md with Sprint 1 defined
- Create REQ-0001 through REQ-000N from your objectives
- Create ARCH-001 with the overall system architecture
- Create agents/todo.md with the full Sprint 1 backlog
- Commit everything with `[Claude Manager]` prefix

### Step 4: Review what was generated

After bootstrap, review:
- `CLAUDE.md` — Does the tech stack and project description look right?
- `agents/requirements/REQ-0001.md` — Are the requirements specific and testable?
- `agents/architecture/ARCH-001.md` — Does the architecture match your vision?
- `agents/todo.md` — Are the Sprint 1 tasks in the right order?

Edit anything that needs adjusting. Commit with `[Claude Manager]` prefix.

### Step 5: Run `/go` to start executing

```
/go
```

The sprint execution loop will:
- Activate the appropriate agents for current sprint tasks
- Programmer implements features per REQ + ARCH documents
- QA Agent writes tests
- Tech Lead reviews at sprint end
- Project Manager closes sprint and opens the next one

Repeat `/go` each session to continue progress.

```
│   ├── todo.md              ← Generated by /bootstrap
│   ├── requirements/
│   │   ├── REQ-0001.md      ← Generated from your objectives
│   │   └── REQ-0002.md

│   │   └── ARCH-001.md      ← Generated by /bootstrap
│   ├── handoffs/            ← Inter-agent handoffs
│   ├── reviews/code/        ← Tech Lead sprint reviews
│   ├── session-logs/        ← Per-agent session logs
│   ├── templates/           ← Document templates
│   └── workflow/            ← Agent role definitions and patterns
└── [your source code]       ← Built by Programmer agent
```

## Commands

| Command | What it does |
|---------|-------------|
| `/bootstrap` | Initialize project from PROJECT.md (run once) |
| `/go` | Run sprint execution loop — evaluate status, activate agents |
| `/plan [topic]` | Planning session for a feature or technical question |

## Sprint Cycle

```
Sprint Kickoff
    │
    ├── Product Manager creates REQ documents
    ├── Software Architect creates ARCH documents
    ├── Programmer implements features
    └── QA Agent writes tests
    │
Sprint End
    │
    └── Tech Lead reviews all sprint output (mandatory)
            │
            ├── Approved → Project Manager closes sprint, opens next
            └── Changes Required → Programmer fixes → TL re-reviews
```

**Sprint Gate Rule**: No sprint closes without Tech Lead Approved or Conditional Approval.

---

## Key Principles

1. **Single source of truth**: PROJECT.md drives everything. REQs drive implementation. ARCHs drive design.
2. **Auditable history**: Git commit prefixes (`[Prog]`, `[Arch]`, etc.) make the agent trail searchable.
3. **Sprint gate**: Tech Lead review is mandatory — quality is not optional.
4. **Handoffs over memory**: Agents communicate through documents, not assumptions.
5. **Complexity-aware**: Simple features skip the architecture step. Complex ones require ADRs.

## Customizing the Workflow

- **Change agent definitions**: Edit `agents/workflow/agents.md`
- **Change code standards**: Edit `agents/workflow/python-standards.md` or add `typescript-standards.md`
- **Change sprint cadence**: Edit the patterns in `agents/workflow/patterns.md`
- **Add slash commands**: Add files to `.claude/commands/`
- **Change document formats**: Edit templates in `agents/templates/`

## Frequently Asked Questions

**Q: Can I use this without Claude Code?**
A: The workflow documents and templates work with any AI assistant. The `.claude/commands/` files are Claude Code-specific slash commands but the prompts inside work anywhere.

**Q: What if my project isn't Python?**
A: Fill in your tech stack in PROJECT.md. The `/bootstrap` command will generate appropriate commands in CLAUDE.md. You may want to add a `typescript-standards.md` or similar for your language.

**Q: How long are sprints?**
A: Up to you. The workflow doesn't enforce a duration. Typically 1-2 weeks. Run `/go` at the start of each session to pick up where you left off.

**Q: What if I'm a solo developer?**
A: The multi-agent workflow works great solo — you activate different agents at different times. Add to PROJECT.md Constraints: "Single developer — keep complexity low."

**Q: Can multiple people use this?**
A: Yes. Each person can run agents independently. The git commit prefixes and handoff documents provide coordination. Consider adding a `CONTRIBUTING.md` that points to `agents/workflow/agents.md`.
