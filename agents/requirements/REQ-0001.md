## Summary

Demonstrate `cuda.core` and `cuda.bindings` APIs through hands-on, runnable Python examples that cover the essential patterns needed for ML workloads: device initialization, memory allocation, kernel compilation, kernel launch, result retrieval, pinned host memory for fast transfers, and loading data directly from disk and network into GPU memory.

---

## Background

CUDA Python exposes two layers: `cuda.bindings` (raw C API bindings) and `cuda.core` (Pythonic high-level wrappers). Developers coming from NumPy or PyTorch may not know which API to use when, or how to set up a minimal CUDA Python program. This demo establishes the foundation all other demos build on.

## Notes

- cuda.core docs: https://nvidia.github.io/cuda-python/cuda-core/latest/
- cuda.bindings docs: https://nvidia.github.io/cuda-python/cuda-bindings/latest/
- Vector-add is the canonical "hello world" for CUDA — a simple element-wise A[i] + B[i] = C[i]
- Pinned memory is allocated via `cudaHostAlloc` (exposed via `cuda.bindings.cudart.cudaHostAlloc`); the resulting buffer can be memory-mapped directly into DMA, bypassing the CPU for transfers
- For disk loading: use `numpy.load()` into a pinned buffer, or `numpy.frombuffer()` on a pinned allocation
- For network loading: `urllib.request` or `requests` to stream bytes into a pinned buffer; wrap in try/except for offline fallback
- Pinned memory must be explicitly freed with `cudaFreeHost` — it is not managed by the Python GC
