## Summary

Add Numba-CUDA and CuPy variants of the core algorithm demos (vector add, k-means, PCA, linear regression, naive Bayes, and custom kernels), giving users a four-way comparison: NumPy CPU / cuda-python GPU / Numba-CUDA GPU / CuPy GPU. Each variant lives as a side-by-side file (`numba_*.py`, `cupy_*.py`) alongside the existing `gpu_*.py` and `cpu_*.py` files.

---

## Background

The existing demos teach cuda-python at the kernel level. Many users arrive from the Numba ecosystem (`@cuda.jit`) or from NumPy-style CuPy code. Seeing the same algorithm expressed three different ways — raw NVRTC kernels (cuda-python), JIT-decorated Python functions (Numba), and NumPy-style array operations (CuPy) — lets users map their existing knowledge onto the GPU programming model and choose the right tool for their project. This is the multi-backend comparison layer that elevates the project from a cuda-python tutorial to a GPU Python reference.

## Notes

- Numba `@cuda.jit` compiles Python to PTX via LLVM — different backend from NVRTC (cuda-python uses NVRTC). Neither is "better"; they target different use cases.
- CuPy's `cp.RawKernel` is closer to cuda-python: it compiles CUDA C strings via NVRTC. `cp.matmul` / `cp.linalg` call into cuBLAS/cuSOLVER. Both are useful to show.
- Numba warmup pattern: call the kernel once on a small array before starting `BenchmarkRunner` timing. The JIT cost is a one-time compile, not a per-call overhead.
- CuPy stream sync: use `cp.cuda.Stream.null.synchronize()` before reading host-side timers to ensure GPU work is complete.
