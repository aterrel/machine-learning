## Summary

Create `src/kernel_model/` — a pure-Python library that analytically models GPU kernel performance without executing any code on a GPU. Given kernel launch parameters and device specs, it computes theoretical occupancy (fraction of warps in flight), roofline performance bound (compute-bound vs memory-bound classification), and predicted throughput. A companion demo `demos/09_kernel_model/` exercises the library and shows how to use it to reason about kernel design trade-offs.

---

## Background

Writing a CUDA kernel is only half the challenge — understanding *why* it runs at a certain speed is the other half. Two fundamental tools for this are the occupancy model (how many warps can the SM keep in flight?) and the roofline model (is the kernel limited by compute or by memory bandwidth?). Both are deterministic analytical calculations that can be performed in pure Python given a handful of kernel parameters and device properties.

This library gives ML practitioners a GPU-free way to explore kernel design trade-offs before writing any CUDA code, and complements the hands-on demos by providing the theoretical framework behind observed speedups.

## API / Interface Design

```python
# src/kernel_model/__init__.py
from .device_spec import DeviceSpec
from .occupancy import OccupancyModel, OccupancyResult
from .roofline import RooflineModel, RooflineResult

# src/kernel_model/device_spec.py
@dataclass
class DeviceSpec:
    name: str
    sm_count: int
    max_warps_per_sm: int
    max_blocks_per_sm: int
    shared_mem_per_sm: int       # bytes
    registers_per_sm: int
    peak_fp32_gflops: float
    memory_bandwidth_gbs: float

    @classmethod
    def from_name(cls, gpu_name: str) -> "DeviceSpec": ...

    @classmethod
    def from_device(cls, device) -> "DeviceSpec": ...  # accepts cuda.core.Device

# src/kernel_model/occupancy.py
@dataclass
class OccupancyResult:
    block_size: int
    warps_per_block: int
    active_blocks_per_sm: int
    active_warps_per_sm: int
    max_warps_per_sm: int
    occupancy: float                      # 0.0–1.0
    limiting_resource: str                # "warps" | "shared_memory" | "registers" | "blocks"

class OccupancyModel:
    def __init__(self, device: DeviceSpec): ...
    def compute(self, block_size: int, shared_mem_per_block: int,
                registers_per_thread: int) -> OccupancyResult: ...
    def sweep_block_sizes(self, shared_mem_per_block: int,
                          registers_per_thread: int,
                          block_sizes: list[int] | None = None) -> list[OccupancyResult]: ...

# src/kernel_model/roofline.py
@dataclass
class RooflineResult:
    arithmetic_intensity: float    # FLOP/byte
    ridge_point: float             # FLOP/byte
    bound: str                     # "compute" | "memory"
    predicted_gflops: float

class RooflineModel:
    def __init__(self, device: DeviceSpec): ...
    def compute(self, flops: float, bytes_accessed: float) -> RooflineResult: ...
    def sweep_intensities(self, intensity_range: list[float]) -> list[RooflineResult]: ...
```

## Out of Scope

- Warp-level or thread-level simulation (functional CPU simulation of CUDA semantics)
- CUDA graph modeling
- Multi-GPU or NVLink bandwidth modeling
- Dynamic shared memory modeling beyond fixed per-block allocation
- FP16/BF16/TF32/INT8 performance modeling (FP32 only for now)

## Notes

- Occupancy calculation follows NVIDIA's occupancy calculator methodology: three limiters (warps, shared memory, registers) determine the binding constraint; the minimum sets active blocks per SM
- Registers per thread must be rounded up to the next multiple of 256 (per CUDA warp register allocation granularity) before computing register pressure
- Roofline model reference: Williams, Waterman, Patterson (2009) — "Roofline: An Insightful Visual Performance Model for Multicore Architectures"
- `DeviceSpec.from_device()` uses `cuda.core.Device.properties` which returns a `DeviceProperties` object with fields like `max_threads_per_block`, `max_shared_memory_per_block`, `regs_per_block`, `warp_size`
- The hardcoded GPU table should use canonical NVIDIA product names as keys (case-insensitive match recommended)
