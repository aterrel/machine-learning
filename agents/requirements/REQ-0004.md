# REQ-0004: CPU vs GPU Benchmarking with Speedup Metrics

**Status**: Active
**Priority**: P0
**Objective Reference**: Objective 4 — Side-by-side comparisons with timing and speedup metrics
**Created**: 2026-05-01
**Author**: Product Manager
**Assigned To**: Programmer

---

## Summary

Provide a unified benchmarking harness that runs CPU and GPU implementations of each demo, records wall-clock and GPU-event timing, computes speedup ratios, validates numerical correctness, and outputs a structured report — both to stdout and as a JSON file.

---

## Background

Showing raw speedup numbers is the primary way to communicate GPU value to ML practitioners. Without a rigorous, reproducible benchmark harness, comparisons are anecdotal. This REQ ensures every demo participates in a consistent measurement framework that accounts for GPU warmup, data-transfer overhead, and statistical variability.

---

## Requirements

### Functional Requirements

| ID | Requirement | Priority |
|----|-------------|---------|
| REQ-0004-F1 | Implement `BenchmarkRunner` that runs each demo's CPU and GPU functions N times and reports mean/min/max | P0 |
| REQ-0004-F2 | Use CUDA events for GPU timing (not Python wall clock alone) to exclude Python overhead | P0 |
| REQ-0004-F3 | Include GPU warmup run (1 iteration discarded) before timing loop | P0 |
| REQ-0004-F4 | Report data transfer time (host-to-device + device-to-host) separately from compute time | P1 |
| REQ-0004-F5 | Output results to stdout in a human-readable table | P0 |
| REQ-0004-F6 | Output results as JSON to `benchmarks/results/YYYY-MM-DD-<demo>.json` | P1 |
| REQ-0004-F7 | CLI tool `python benchmarks/run_all.py` runs all demos and prints a summary table | P0 |
| REQ-0004-F8 | Correctness check (output delta vs CPU) is always reported alongside timing | P0 |

### Non-Functional Requirements

| ID | Requirement | Priority |
|----|-------------|---------|
| REQ-0004-NF1 | Benchmark harness overhead must not exceed 5% of measured runtime | P0 |
| REQ-0004-NF2 | JSON output must be parseable and stable across runs for the same seed and hardware | P1 |
| REQ-0004-NF3 | Must handle CUDA OOM gracefully (report error, continue to next benchmark) | P0 |

## Acceptance Criteria

- [ ] AC-1: User can run `python benchmarks/run_all.py` and sees a table with Algorithm, CPU time, GPU time, Speedup, Correct columns for all demos
- [ ] AC-2: GPU timing uses `cuda.core` event records — verified by inspecting source code
- [ ] AC-3: Running the same benchmark twice with the same seed produces timing within 10% variance
- [ ] AC-4: A demo that produces incorrect GPU output reports `Correct: False` and flags it visually in output
- [ ] AC-5: Benchmark JSON file is written to `benchmarks/results/` and contains all required fields
- [ ] AC-6: OOM scenario (set artificially via very large n_samples) prints error row and continues

## API / Interface Design

```python
# benchmarks/benchmark_runner.py
@dataclass
class BenchmarkResult:
    demo_name: str
    n_samples: int
    n_features: int
    cpu_time_mean_s: float
    cpu_time_min_s: float
    gpu_time_mean_s: float      # CUDA event time, compute only
    gpu_transfer_s: float       # H2D + D2H time
    speedup: float              # cpu_time_mean / gpu_time_mean
    speedup_with_transfer: float
    max_abs_error: float
    correct: bool               # max_abs_error < tolerance

class BenchmarkRunner:
    def run(self, demo_fn, n_repeats: int = 5, **demo_kwargs) -> BenchmarkResult: ...

# CLI:
# python benchmarks/run_all.py [--n-samples INT] [--n-repeats INT] [--output-dir PATH]
```

## Notes

- GPU warmup is critical — first kernel launch incurs JIT/driver overhead not representative of steady-state
- CUDA event timing is more accurate than Python `time.perf_counter` for short GPU kernels
- Consider using `cupy.cuda.Event` or `cuda.bindings.cudart` events for timing
