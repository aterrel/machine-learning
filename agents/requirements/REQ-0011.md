## Summary

Implement `src/kernel_model/ptx_tracer.py` — a pure-Python PTX instruction tracer that scans a PTX source string, classifies every instruction into a hardware-relevant category (smem, tmem, MMA, async copy, fused FP, global mem), and produces an annotated `TracerResult` with per-category counts, architecture-specific latency/throughput annotations, and a bottleneck classification. A companion demo `demos/10_ptx_tracer/` shows the tracer applied to a real kernel from the existing demo suite.

---

## Background

The roofline and occupancy models in REQ-0010 answer "is this kernel compute-bound or memory-bound?" at a high level. The PTX tracer answers a more specific question: *what is the instruction mix of this kernel, and which hardware resource does each instruction class stress on a given architecture?*

This matters because modern GPU architectures have radically different execution units and instruction scopes:
- Ampere/Ada: warp-level MMA (`mma.sync`, 32 threads), accumulator in RF
- Hopper: warpgroup-level MMA (`wgmma.mma_async`, 128 threads), accumulator in RF
- Blackwell: CTA-level MMA (`tcgen05.mma`, issued by 1 thread), accumulator in TMEM (256 KB/SM)

The same PTX (if recompiled per architecture) will produce different instruction mixes and different bottlenecks depending on the target SM. The tracer surfaces these differences analytically, without running any GPU code.

## API / Interface Design

```python
# src/kernel_model/ptx_tracer.py

@dataclass
class InstructionRecord:
    mnemonic: str
    category: str          # one of the 10 categories
    arch_introduced: str   # "sm_80" | "sm_89" | "sm_90" | "sm_100"
    latency_cycles: dict[str, int | None]    # sm_version → cycles
    throughput_note: str   # human-readable throughput annotation

@dataclass
class TracerResult:
    arch: str                              # DeviceSpec.name
    sm_version: str                        # "sm_80" | "sm_90" | ...
    total_instructions: int
    by_category: dict[str, int]            # category name → count
    mma_warp_count: int                    # mma.sync instructions
    mma_warpgroup_count: int               # wgmma.mma_async instructions
    mma_cta_count: int                     # tcgen05.mma instructions
    smem_alloc_bytes: int | None           # from .shared declarations; None if not found
    tmem_alloc_bytes: int | None           # from tcgen05.alloc; None if not Blackwell
    async_copy_groups: int                 # cp.async.commit_group + bulk equiv
    bottleneck: str                        # "compute" | "memory" | "latency" | "mixed"
    arithmetic_intensity: float | None     # estimated FLOP/byte; None if insufficient data
    unsupported_instructions: list[str]    # instructions not valid on target arch
    notes: list[str]                       # human-readable observations

class PTXTracer:
    def __init__(self, device: DeviceSpec): ...

    def trace(self, ptx_source: str) -> TracerResult: ...

    def trace_file(self, path: str) -> TracerResult: ...

    def bottleneck(self, result: TracerResult) -> str: ...


# Taxonomy and arch table are module-level dicts:
_INSTRUCTION_TAXONOMY: dict[str, InstructionRecord]   # key: mnemonic prefix
_ARCH_TABLE: dict[str, ArchSpec]                       # key: "sm_80" etc.
```

## Instruction Category Definitions

| Category | PTX mnemonic prefixes | SM range |
|---|---|---|
| `smem` | `ld.shared`, `st.shared`, `ldmatrix` | sm_75+ |
| `global_mem` | `ld.global`, `st.global` | all |
| `async_copy` | `cp.async` (non-bulk) | sm_80+ |
| `async_tma` | `cp.async.bulk`, `cp.reduce.async.bulk`, `multimem.cp.async.bulk` | sm_90+ |
| `mma_warp` | `mma.sync`, `mma.sp.sync` | sm_70–sm_89 |
| `mma_warpgroup` | `wgmma.mma_async`, `wgmma.fence`, `wgmma.commit_group`, `wgmma.wait_group` | sm_90+ |
| `mma_cta` | `tcgen05.mma`, `tcgen05.mma.sp`, `tcgen05.mma.ws` | sm_100+ |
| `tmem` | `tcgen05.alloc`, `tcgen05.dealloc`, `tcgen05.ld`, `tcgen05.st`, `tcgen05.cp`, `tcgen05.shift`, `tcgen05.fence`, `tcgen05.commit`, `tcgen05.wait`, `tcgen05.relinquish_alloc_permit` | sm_100+ |
| `fused_fp` | `fma.rn`, `mad.lo`, `mad.hi`, `dp4a` | all |
| `other` | everything else | — |

## MMA Tile Sizes and Warp Scope (for throughput annotation)

| Category | Arch | Warp scope | Tile (M×N×K) example (f16→f32) |
|---|---|---|---|
| `mma_warp` | sm_80–sm_89 | 1 warp (32 threads) | 16×8×16 |
| `mma_warpgroup` | sm_90 | 1 warpgroup (128 threads) | 64×N×16 (N=8..256) |
| `mma_cta` | sm_100 | 1 thread issues; CTA executes | 64×64..128×256 |

## Out of Scope

- Cycle-accurate warp-scheduling simulation (modeled analytically, not simulated)
- SASS-level analysis (PTX only; SASS is post-ptxas and not portable)
- Instruction reordering by `ptxas` (tracer models PTX intent, not final SASS schedule)
- Multi-kernel or multi-SM analysis
- Blackwell 2SM (`tcgen05.mma.2sm`) cluster modeling
- Dynamic shared memory size inference (caller must pass if known)

## Notes

- Research doc: `agents/architecture/ptx-tracer-research.md` contains the full instruction taxonomy, latency tables, architecture throughput data, and existing parser survey used to define this REQ
- The regex approach for PTX scanning is `r'^\s+([\w.]+)'` to extract the leading mnemonic, followed by a prefix lookup in `_INSTRUCTION_TAXONOMY`
- PTX mnemonic lookup must use prefix matching (longest prefix wins): `tcgen05.mma.sp.sync` should match `tcgen05.mma.sp`, not just `tcgen05`
- The `wgmma.fence`, `wgmma.commit_group`, and `wgmma.wait_group` management instructions should be counted in `mma_warpgroup` to give a complete picture of wgmma usage overhead
- Blackwell tmem allocation in bytes: `n_cols × 128 lanes × 4 bytes/lane = n_cols × 512`
- `pyptx` (github.com/patrick-toulme/pyptx) achieves byte-identical PTX round-trips on 218+ real PTX files from CUTLASS/Triton/ThunderKittens — use as parser reference if regex proves insufficient for complex PTX
